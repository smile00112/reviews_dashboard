# Design — Страница «Отзывы» по прототипу GeoMonitor

Дата: 2026-07-18
Источник: `docs/plans/dashboard_new/GeoMonitor — SERM Dashboard Prototype.html` (секция `#screen-reviews`, строки 1833–2151).
Статус: утверждено пользователем (объём «UI + триаж»).

## Цель

Заменить текущую таблицу `/reviews` на страницу из прототипа: лента карточек отзывов
со статус-табами, вторичными фильтрами, внутренним триаж-workflow (статус, флаг
«покупной») и панелью аспектного анализа.

Публикация ответов на площадках остаётся вне объёма (конституция: read-only).
Триаж — внутренние поля БД (`status`, `is_paid`, `paid_cost` уже существуют,
feature 004), меняются только в нашей БД.

## Решения (утверждены)

1. **Объём** — полный UI прототипа + внутренний триаж (PATCH под админ-auth).
   Без кнопок «Ответить», «Назначить»; без заказов, жалоб, заметок.
2. **«Шаблоны ответов»** — пропустить полностью (кнопки нет).
3. **«Группировать по площадкам»** — пропустить (фильтр по площадке покрывает).
4. **Архитектура** — композиция эндпоинтов, не один толстый feed-эндпоинт.

## Backend

### 1. Расширение `GET /api/reviews` (существующий)

Новые query-параметры (все опциональны, комбинируются с существующими
`organization_id`, `rating`, `date_from`, `date_to`, `new_only`, `limit`, `offset`):

| Параметр | Значения | Семантика |
|---|---|---|
| `status_tab` | `all` \| `unanswered` \| `in_progress` \| `escalated` \| `answered` | `answered` = `response_text IS NOT NULL`; `unanswered` = `response_text IS NULL`; `in_progress`/`escalated` = по полю `status`. Табы могут пересекаться — это фильтры, не разбиение. |
| `platform` | `yandex` \| `google` \| `gis2` | по полю `platform` |
| `tone` | `neg` \| `pos` | `neg` = rating 1–3, `pos` = rating 4–5 (как в прототипе, по звёздам) |
| `period` | `24h` \| `7d` \| `30d` \| `year` | по `review_date`; для отзывов без `review_date` — по `date(first_seen_at)`. `24h` = «за последние 2 календарных дня» (точность даты — день). |
| `is_paid` | bool | флаг «покупной» |
| `aspect` | строка-категория | `problems` JSONB содержит запись с `category == aspect` |
| `sort` | `new` (default) \| `criticality` | `new` = `review_date DESC NULLS LAST, first_seen_at DESC`. `criticality` = без ответа первыми, затем рейтинг ASC, затем новизна DESC. |

В `ReviewResponse` добавить `status`, `is_paid`, `paid_cost`, `platform` —
поля уже в модели, в схему не экспортированы.

### 2. `GET /api/reviews/summary` (новый)

Принимает те же фильтры, что и список, **кроме** `status_tab` (счётчики считаются
поверх остальных фильтров). Возвращает:

```json
{
  "total": 7706,
  "new_count": 23,
  "unanswered": 210,
  "in_progress": 14,
  "escalated": 3,
  "answered": 7479,
  "overdue_24h": 12,
  "negative": 114
}
```

- `new_count` = отзывы со статусом `new` (не просмотрены) — как текущее `new_only`.
- `overdue_24h` = без ответа и `first_seen_at` старше 24 часов.
- `negative` = rating 1–3 (бейдж чипа «Негатив»).

Один запрос с условными агрегатами (`count(*) FILTER (WHERE ...)` /
эквивалент через `case`), не 7 запросов.

### 3. `GET /api/reviews/aspects` (новый)

Параметры: `period` (как выше, default `30d`), `organization_id`, `platform`,
`aspect` (для тренда). Возвращает:

```json
{
  "aspects": [
    {
      "category": "delivery",
      "label": "Доставка",
      "mentions": 892,
      "delta_pct": 42,          // vs предыдущий период той же длины; null если prev=0
      "pos": 14, "neu": 21, "neg": 65   // доли в % по sentiment отзывов с этим аспектом
    }
  ],
  "trend": {                     // только если задан ?aspect=
    "category": "delivery",
    "days": 90,
    "series": [ {"date": "2026-04-19", "count": 3}, ... ]
  }
}
```

- Источник — `problems` JSONB (существующая 8-категорийная таксономия анализа,
  константные русские лейблы из `analysis/problems.py`). Никаких LLM.
- Для `period=year` сравнительный период — предыдущий год; тренд всегда 90 дней.
- Агрегация в `AnalysisService` (или рядом), выборка отзывов с `problems IS NOT NULL`
  за окно, разбор в Python (объёмы — тысячи строк, приемлемо; JSONB-операторы
  не используем, чтобы SQLite-тесты работали).

### 4. `PATCH /api/reviews/{id}` (новый, admin-auth)

Тело (все поля опциональны): `status` (`new|in_progress|answered|escalated`),
`is_paid` (bool), `paid_cost` (int, null сбрасывает). Возвращает обновлённый
`ReviewResponse`. Auth — как остальные мутации (существующая админ-зависимость).
Полей дедуп-хэша не касается. 404 если отзыв не найден.

### 5. Тесты (критический путь)

- `test_reviews_filters.py`: каждый `status_tab`, `tone`, `period`, `aspect`,
  `sort=criticality`, комбинация фильтров.
- summary: счётчики согласованы с фильтрами, `overdue_24h`.
- aspects: mentions/delta/sentiment-доли на фикстурах, пустая БД → пустой список.
- PATCH: смена статуса, is_paid, 401/403 без auth, 404.

## Frontend

Переписать `app/(dashboard)/reviews/page.tsx`. Компоненты — `components/reviews/`.
Все фильтры живут в URL `searchParams` (диплинки из «Требуют внимания» на
`/reviews?rating=1&...` продолжают работать; `new_only` поддержать).

### Структура

- `page.tsx` — клиентская страница (как сейчас): читает searchParams, грузит
  `summary` + список + аспекты, управляет состоянием фильтров через router.replace.
- `status-tabs.tsx` — крупные табы со счётчиками из summary (`danger`-подсветка
  для «Не отвечено» и «Эскалированные»).
- `review-filters.tsx` — чипы: тональность (Все/Негатив 1–3★ с бейджем/Позитив 4–5★),
  период (24ч/7д/30д/Год), площадка (Все/Я/G/2Г), организация (select),
  «💎 Покупные», сброс.
- `review-card.tsx` — карточка: автор, организация, тег площадки, относительное
  время, звёзды (красные при 1–3), текст, статус-бейдж («БЕЗ ОТВЕТА · Nч» /
  «В РАБОТЕ» / «🔥 ЭСКАЛИРОВАНО» / «ОТВЕЧЕН»), тональность (score при наличии),
  кликабельные аспект-теги (→ `?aspect=`), блок ответа компании (display-only),
  действия: «В работу» / «Эскалировать»↔«Снять эскалацию», тумблер «💎 Покупной»
  (+ инлайн-инпут стоимости при включении). Оптимистичное обновление, откат при
  ошибке PATCH.
- `aspects-panel.tsx` — таблица аспектов с CSS sentiment-барами, Δ-бейджами
  (`▲`/`▼`/`—`), клик по строке → `?aspect=`; под таблицей легенда и мини-тренд
  (рукописный SVG-полилайн, как донаты в overview) для выбранного аспекта.
- Empty-state прототипа (иконка, текст, кнопка «Сбросить фильтры»).
- Пагинация — кнопка «Показать ещё» (offset+=limit, аппенд).
- Шапка: заголовок + сабтайтл «N отзывов · M новых · K без ответа · L просрочены > 24ч».

### API-клиент

`lib/api.ts`: расширить `listReviews` параметрами, добавить `getReviewsSummary`,
`getReviewAspects`, `patchReview`. `lib/types.ts`: `ReviewSummary`, `AspectStat`,
`AspectTrend`, расширить `Review` (`status`, `is_paid`, `paid_cost`, `platform`).
PATCH шлёт креды как остальные админ-мутации.

### Стиль

Тёмная тема и токены уже введены overview-страницей — переиспользовать классы/
паттерны оттуда (панели, чипы, бейджи). Ничего нового в зависимостях.

### E2E

`apps/web/tests/reviews.spec.ts` по образцу `overview.spec.ts`: страница
рендерится, табы/чипы меняют URL и перезапрашивают, карточка показывает ответ
компании, смена статуса требует залогиненного админа (гейт как в overview-suite).

## Вне объёма

Кнопки «Ответить», «Шаблоны ответов», «Назначить»; привязка заказов; жалобы
площадке; заметки; группировка по площадкам; мультиселект локаций (одиночный
select организации); публикация чего-либо на внешние площадки.

## Порядок сборки

1. Схемы + сервис: фильтры списка, summary, aspects, PATCH + тесты (`pytest -v`).
2. `lib/types.ts` + `lib/api.ts`.
3. Компоненты + страница.
4. E2E + `npm run lint` + `tsc`.
