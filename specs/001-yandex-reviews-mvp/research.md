# Research: Yandex Reviews MVP

**Date**: 2026-06-14

## R-001: Scrape Orchestration

**Decision**: Use FastAPI `BackgroundTasks` for MVP scrape execution (no Celery/Redis).

**Rationale**: Constitution principle V (YAGNI). Internal tool with manual triggers and
tens of organizations does not need a distributed queue yet. Background tasks provide
async execution with minimal infrastructure.

**Alternatives considered**:
- Celery + Redis — rejected for MVP complexity and ops overhead
- Synchronous scrape in request handler — rejected; would block HTTP and timeout on long scrapes
- Cron-only scheduling — rejected; spec requires manual triggers only

## R-002: Browser Automation Stack

**Decision**: Python Playwright in `apps/api` scraper module, headless Chromium in production,
headed mode optional for local operator login debugging.

**Rationale**: Playwright has mature storage-state support for session persistence, good
Python integration with FastAPI stack, and screenshot/HTML capture for debug artifacts.

**Alternatives considered**:
- Node Playwright in separate service — rejected; adds process boundary and deployment unit
- Selenium — rejected; weaker storage state ergonomics
- HTTP-only scraping without browser — rejected; Yandex Maps reviews require JS-rendered DOM

## R-003: Review Deduplication Strategy

**Decision**: Unique constraint on `(organization_id, content_hash)` where `content_hash` is
SHA-256 of normalized `author_name | rating | review_date_text | review_text`.

**Rationale**: Yandex may not expose stable external review IDs on public pages; content-based
hash with normalization handles whitespace variance. Aligns with reference plan section 3.

**Alternatives considered**:
- External review ID only — rejected; often unavailable on public scrape
- Database natural key on author+date without hash — rejected; text edits would duplicate

## R-004: Authentication Model

**Decision**: No application-level auth for MVP. Yandex operator credentials via
`YANDEX_OPERATOR_LOGIN` / `YANDEX_OPERATOR_PASSWORD` env vars; Playwright `storage_state`
saved to `YANDEX_STORAGE_STATE_PATH`.

**Rationale**: Internal trusted-network tool per spec assumptions. Constitution explicitly
excludes app auth.

**Alternatives considered**:
- Basic auth on dashboard — deferred post-MVP
- OAuth for Yandex — rejected; operator password + storage state is simpler for internal use

## R-005: Frontend Stack

**Decision**: Next.js App Router, TailwindCSS, shadcn/ui components, fetch-based API client
in `apps/web/lib/api.ts`.

**Rationale**: Matches reference plan; shadcn provides accessible tables/forms quickly for
internal dashboard. No SSR requirement for MVP — client-side fetch acceptable.

**Alternatives considered**:
- Vite + React SPA — viable but reference plan specifies Next.js
- Server Components for all data — rejected for MVP; simpler client fetch with refresh buttons

## R-006: Failure Handling for Access Challenges

**Decision**: Map captcha, 2FA, and manual confirmation to scrape/session status
`needs_manual_action`; save screenshot + HTML snapshot; do not implement anti-captcha.

**Rationale**: Constitution and spec FR-019 prohibit forced bypass. Operators resolve
manually (headed login, refresh session).

**Alternatives considered**:
- Retry with exponential backoff only — insufficient; does not signal manual intervention
- Third-party captcha solver — explicitly out of scope

## R-007: Scraper Operational Limits

**Decision**:
- Max page load wait: 30 seconds
- Max review panel scrolls: 40 per organization
- Scroll delay: 800–1500 ms random between scrolls
- Debug output directory: `SCRAPER_DEBUG_DIR` (default `.local/scraper-debug`)

**Rationale**: Prevents runaway scroll loops; balances completeness vs. timeout for MVP.

**Alternatives considered**:
- Unlimited scroll until end — rejected; risk of infinite loop on dynamic pages
- Fixed 10-scroll limit — rejected; may miss reviews on popular locations

## R-008: Database & Migrations

**Decision**: PostgreSQL 16 via Docker Compose; SQLAlchemy 2.x models; Alembic migration
`0001_initial.py` for all four entities.

**Rationale**: Standard Laravel-adjacent stack familiarity; UUID PKs for distributed-friendly IDs.

**Alternatives considered**:
- SQLite for MVP — rejected; concurrent scrape writes and production parity favor Postgres

## R-009: API Style

**Decision**: REST JSON under `/api/*` prefix; OpenAPI auto-generated by FastAPI.

**Rationale**: Simple contract for Next.js client; matches reference plan section 4.

**Alternatives considered**:
- GraphQL — rejected; overkill for CRUD + trigger endpoints

## R-010: Organization Delete Semantics

**Decision**: Hard delete organizations for MVP (cascade or explicit delete of reviews/runs).

**Rationale**: Spec assumption states hard delete acceptable; no compliance soft-delete requirement.

**Alternatives considered**:
- Soft delete with `deleted_at` — deferred unless compliance need emerges
